Principe opérationnel 80/20 : démarrer avec les 80% de données valides, traiter les anomalies au cas par cas. Ne pas attendre 100% de fiabilisation pour lancer — c'est une règle actée en Discovery pour ne pas bloquer le delivery.

Étapes ETL détaillées

①

Ingestion

Connecteurs API (AS24, Factorial, Masternaut) + import CSV automatisé (Engie, Dirham). Fréquence par source selon priorité.

②

Validation

Contrôles d'intégrité : formats, champs obligatoires, plages de valeurs. Rejet vers file d'exceptions si anomalie.

③

Normalisation

Conversion des formats (ex. virgules CSV Engie → points), standardisation des immatriculations (espaces, tirets).

④

Mapping

Correspondance entre entités externes et référentiels internes (immatriculations, conducteurs, contrats, clients).

⑤

Dédoublonnage

Détection des transactions dupliquées (ex. plein AS24 apparaissant 2x). Règle de conservation la plus récente.

⑥

Logs & alertes

Traçabilité complète + alertes automatiques sur anomalies critiques (ex. véhicule avec coût = 0, plein > 2× moyenne).

7.6 · Règles de gouvernance data

Deux patterns pour garder la maîtrise.

Pattern 1

Source-of-Truth Pattern

Chaque type de donnée a une et une seule source de vérité :

  • Paie (Factorial) = coûts RH réels
  • Télématique (Solides / Michelin) = réel terrain (km, conso, amplitude)
  • Planning / TMS = théorique
  • ADV = CA théorique

Rapprochements automatisés journaliers entre les sources — évite les conflits et les versions contradictoires.

Pattern 2

Exception Management 80/20

Règles claires de traitement automatique vs manuel des anomalies :

  • 80% des cas stables → pipeline automatique
  • 20% des exceptions → file de traitement manuel
  • Seuils d'alerte + owner par catégorie
  • SLA de correction des exceptions

Principe pragmatique : lancer avec 80% de données valides, traiter les anomalies au cas par cas plutôt qu'attendre la perfection.
